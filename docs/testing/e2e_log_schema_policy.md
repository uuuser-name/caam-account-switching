# E2E Log Schema Versioning Policy

This policy defines compatibility rules for `docs/testing/e2e_log_schema.json`.

## Versioning

- Schema follows semantic versioning as `schema_version`.
- `MAJOR`: breaking changes (field removals, type changes, required-set changes).
- `MINOR`: backward-compatible additive changes (new optional fields, metadata).
- `PATCH`: clarifications and non-behavioral updates.

## Compatibility Rules

- Consumers must accept all events valid under the same `MAJOR` version.
- `MINOR` updates must be additive-only and must not invalidate prior valid events.
- Breaking changes require:
1. A migration note in this file.
2. Validator updates with fixtures for old/new behavior.
3. A bead comment linking verification evidence.

## Change Workflow

1. Update schema and `schema_version`.
2. Update validator fixtures and run `make validate-e2e-schema`.
3. Run `make test-audit` to regenerate artifacts.
4. Document rationale and compatibility impact in bead evidence comments.

## Validator Contract (Current)

`scripts/validate_e2e_log_schema.sh` enforces both field-level schema constraints and run-level integrity checks:

1. Required-field/type validation and redaction deny-pattern checks.
2. Decision allowlist validation (`pass|continue|retry|abort`).
3. Single-run/single-scenario consistency per JSONL file.
4. Timestamp monotonicity (non-decreasing event order).
5. Step accounting: `*-start` / `*-end` pairs must stay balanced across the run; stray ends and duplicated starts that reuse one end are invalid.

## Forensics Packet Contract

Failure triage artifacts are generated by `scripts/generate_failure_packet.sh` and must include:

1. A machine-checkable bundle manifest (`failure_packet_manifest.json`) that conforms to `docs/testing/failure_packet_bundle_schema.json`.
2. Raw copied logs (`raw/*.jsonl`), deterministic merged timeline (`merged_timeline.jsonl`), and concise summary (`FAILURE_SUMMARY.md`).
3. Grouped machine-readable signatures (`failure_index.json`) and concise first-failure narratives (`failure_narratives.md`).
4. Optional schema validation on inputs (enabled by default; disable only with `FAILURE_PACKET_VALIDATE_SCHEMA=0`).

Fixture verification command:

```bash
make validate-failure-packet
```

## Dry-Run / Live-Run Parity Contract

`scripts/verify_e2e_dry_live_parity.sh` enforces transition-level parity between dry-run and live-run canonical logs:

1. Same event count and ordering.
2. Same transition semantics per event (`step_id`, `component`, `decision`, `error.present`, `error.code`).
3. Schema validation is required by default before parity comparison.

Fixture verification command:

```bash
make validate-e2e-parity
```
